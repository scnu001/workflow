# Stage 9: Post-Launch Feedback

## Purpose
Track whether the product actually gets signal. If no launch happened, record that and explain why.

## Required Input Files
- Stage 8 `next-stage-brief.md`
- launch links if any
- metrics sources if any

## Required Substages
1. `01_metrics_collection.md` - Metrics Collection
2. `02_issue_feedback_review.md` - Issue Feedback Review
3. `03_distribution_feedback_review.md` - Distribution Feedback Review
4. `04_user_comment_synthesis.md` - User Comment Synthesis
5. `05_continue_pivot_kill_decision.md` - Continue / Pivot / Kill Decision

## Required Output Files
- all required substage files
- `feedback-summary.md`
- `metrics-summary.md`
- `continue-pivot-kill.md`
- `stage-report.md`
- `checkpoint.md`
- `decision-log.md`
- `next-stage-brief.md`

## Substage Quality Rule
Each completed feedback substage must follow `templates/substage-report-template.md`, contain at least 500 meaningful words, and distinguish real metrics from missing, simulated, or pending feedback.

## Handoff Rule
The stage is incomplete until `next-stage-brief.md` summarizes metrics, feedback, confidence, blockers, and the continue/pivot/pause/kill recommendation for Reflection.

## Failure / Blocker Behavior
If no launch happened or metrics are unavailable, record the blocker clearly. Do not fabricate traction.

## Evidence-First Rule
Feedback substages must cite actual metrics, comments, issues, channel responses, local files, or access failures. If feedback is unavailable, write `Evidence unavailable. This is a hypothesis, not a finding.` Do not simulate traction as evidence.
